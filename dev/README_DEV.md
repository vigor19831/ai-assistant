# dev/ — Developer & AI Workspace

Эта папка содержит всё, что нужно для разработки, тестирования и работы с AI-ассистентом. **Не трогай вручную** файлы, помеченные `[AUTO]` — они генерируются скриптами.


## 📐 Золотые стандарты архитектуры (Архитектурный манифест)

Проект спроектирован с расчетом на максимальную гибкость, долгую жизнь (чистый код) и легкую миграцию на любые технологии. Каждый новый модуль обязан строго следовать этим 7 правилам:

1. **Гексагональная архитектура (Порты и Адаптеры):** Логика приложения (Ядро) полностью изолирована от внешнего мира. Все внешние зависимости (БД, API, файловая система) общаются с Ядром строго через абстрактные интерфейсы (Порты).
2. **Конфигурация отдельно от кода:** Никаких зашитых URL, токенов, путей или портов в коде. Все настройки управляются исключительно через `config.yaml` или переменные окружения (`.env`).
3. **Неприкосновенность Ядра (Sacred Core):** Изменения в Ядре производятся под строжайшим контролем. Ядро эволюционирует только для улучшения бизнес-модели, но никогда не подстраивается под прихоти внешних библиотек или API.
4. **Тест-покрытие (Тестируемость как метрика):** Архитектура должна позволять тестировать Ядро без запуска реальных баз данных или тяжелых ИИ-движков — с помощью легковесных Mock-заглушек.
5. **Полная независимость от LLM-провайдера:** Общение с нейросетями идет через универсальный API-интерфейс. Переключение между `llama-server.exe`, `Ollama`, `vLLM` или `OpenAI` должно происходить заменой одной строки в конфиге.
6. **Агностический UI (Фронтенд через OpenAI API эмуляцию):** Бэкенд не привязан к интерфейсу пользователя. Вывод чата/UI всегда сторонний (браузер, HTML, плагин) и общается с бэком по стандартизированному протоколу OpenAI API.
7. **Разделение контуров Dev и Prod:** Четкая изоляция сред. Папка разработчика (`dev/`) со скриптами, тестами, лаунчерами и утилитами генерации контекста полностью отделена от чистого продакшн-кода приложения (`src/`).
8. **Идемпотентность и отказоустойчивость (Resilience):** Все операции (особенно запись в БД и внешние API-запросы) проектируются безопасными для повторного вызова с теми же данными. Любой внешний сетевой вызов обязан иметь жесткий таймаут и стратегию умных повторов (retries с экспоненциальной задержкой) на случай мимолетных сбоев сети.
9. **Единая сквозная обработка ошибок (Error Mapping):** Адаптеры перехватывают специфичные ошибки внешних библиотек (например, `httpx.TimeoutException` или ошибки СУБД) и оборачивают их в кастомные системные исключения Ядра. Бизнес-логика ничего не знает о сетевых или инфраструктурных ошибках и работает только со своими типами исключений.
10. **Безопасное выключение (Graceful Shutdown):** Приложение умеет правильно завершать работу по сигналам ОС (например, Ctrl+C). При остановке система не обрывает процессы на полуслове, а сначала останавливает прием новых запросов, дожидается выполнения текущих активных задач, корректно закрывает соединения с БД/серверами и только затем выключается без потери данных.

## 🔑 Ключевые файлы

### `AI_RULES.md` — Конституция проекта
**Что это:** Правила, которые AI читает перед каждым ответом.  
**Как попадает в чат:** `context_build.py` автоматически вставляет его в `context_build_compact.md`.  
**Когда редактировать:** Когда меняешь архитектурные принципы (например, разрешил Pydantic в core).  
**Разделы:**
- Sacred Core Policy — что можно менять в `core/`, а что нет
- Red Flags — когда AI должен остановиться и предложить core-изменение
- Adapter Discipline — правила для адаптеров
- Feature Isolation — запрет на cross-feature импорты
- Solo Project Guardrails — **что НЕ делать** (Redis, Celery, ленивый AppState и т.д.)

### `ERROR_TAXONOMY.md` — Карта ошибок `[AUTO]`
**Что это:** Таблица всех исключений в проекте (raise, except, :raises:).  
**Как формируется:** Автоматически скриптом `error_taxonomy_build.py`.  
**Зачем нужен:** AI проверяет таблицу перед тем, как предложить try/except — не добавляет лишний retry туда, где уже есть `with_retry`, не ловит `ValueError` там, где ожидается `AdapterError`.  
**Когда обновляется:** При каждом запуске `context_build.py` (автоматически).  
**Не редактировать руками** — изменения перезапишутся при следующей генерации.

### `TODO.md` — Список задач
**Что это:** Единый список задач по фазам (P0, P1, P2).  
**Как использовать:** Копируешь один пункт, вставляешь в новый чат с AI, получаешь патч.  
**Структура:**
- Фаза 0 — быстрая гигиена (ruff, CI, pre-commit)
- Фаза 1 — защита Sacred Core (frozen PipelineData, config extra, классы-шаги)
- Фаза 2 — API и фичи (versioning, разделение handlers)
- Фаза 3 — данные и observability (trace_id, миграции БД)
- Фаза 4 — документация и ADR

### `pyproject.toml` (в корне) — Единый манифест проекта
**Где лежит:** `D:\ai\pyproject.toml` (не в `dev/`).  
**Структура:**
- `[project]` — runtime-зависимости (fastapi, uvicorn, pydantic, sqlalchemy)
- `[project.optional-dependencies]` — extras:
  - `dev` — pytest, ruff, mypy, mutmut, hypothesis, vulture
  - `faiss` — faiss-cpu (для векторного поиска)
- `[tool.ruff]`, `[tool.mypy]`, `[tool.pytest.ini_options]` — конфиги инструментов

### `launcher.py` — Интерактивный меню
**Что делает:** Два столбца — скрипты слева, тесты справа. Вводишь номер — запускается.  
**Фичи:**
- Зелёная `*` — последний запущенный пункт
- `[bg]` — фоновый процесс (start.py)
- Автоподстановка флагов (`--clean`, `--full`) с подтверждением [y/n]
- Открытие терминала с активированным venv
- **Двойной Enter** — выход из лаунчера + авто-остановка сервера
- **Авто-логи тестов** — полный вывод pytest с трейсбэком сохраняется в `dev/tests_run_YYYY-MM-DD_HH-MM-SS.log`
- **`--no-menu`** — non-interactive режим для CI: `python dev/launcher.py --no-menu 5`
- Все dev-скрипты запускаются только через .venv. Системный Python не поддерживается

## 🆘 Если что-то пошло не так

**`context_build.py` падает с ошибкой:**
- Проверь, что `README.md` и `pyproject.toml` есть в корне проекта (`D:\ai\`)
- Проверь, что запускаешь из корня: `cd D:\ai && python dev/scripts/context_build.py`

**`ERROR_TAXONOMY.md` не обновляется:**
- Проверь, что `error_taxonomy_build.py` лежит рядом с `context_build.py` в `dev/scripts/`
- Запусти вручную: `python dev/scripts/error_taxonomy_build.py`

**AI не видит свежие правила из `AI_RULES.md`:**
- Убедись, что запустил `context_build.py` после правки
- Проверь дату в начале `context_build_compact.md` — должна быть свежей

**TODO.md разросся:**
- Переноси выполненные пункты в раздел `## Done` внизу файла
- Или создавай `TODO-2026-05.md`, `TODO-2026-06.md` по месяцам

    # полный вывод всех ошибок текстов в файл на windows:
pytest dev/tests/ --tb=long -v > test_failures_full.txt 2>&1
pytest dev/tests -vvs --tb=long > test_failures.txt 2>&1
pytest dev/tests -vvs --tb=long --show-capture=all > test_failures.txt 2>&1
    # полный вывод всех ошибок тестов в файл на unix:
pytest dev/tests/ --tb=long -v 2>&1 | tee test_failures_full.txt

```
